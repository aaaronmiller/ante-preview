## 🔴 HIGH Severity

### 1. `unwrap()` on Mutex::lock() in production session code path

**Files**: `crates/agent-sdk/src/sessions/mod.rs`  
**Lines**: 339, 343, 397, 410, 435, 436, 443, 625, 633, 644

Every session write/read path calls `.unwrap()` on `Mutex::lock()`. If a panic occurs in any task holding the lock, the mutex becomes poisoned and every subsequent `.unwrap()` **panics the entire runtime** instead of recovering.

**Suggested fix**: Replace with `lock_or_recover()` pattern (already available in `crates/exec/src/lib.rs:25-30`) or use `.lock().unwrap_or_else(|e| e.into_inner())`.

---

### 2. `#[allow(dead_code)]` on entire `AgentContext` — core runtime fields are wired but unused

**File**: `crates/ante/src/main.rs:298`  
**Lines**: 298-313

The entire `AgentContext` struct is annotated `#[allow(dead_code)]`. This means the following fields are **declared, initialized, but never read** in the active code paths (`run_repl_with_options`, query mode):

| Field | Created | Actually Used? |
|---|---|---|
| `event_bus` | ✅ HookRegistry + EventBus | ❌ — never emits or receives events |
| `mcp_registry` | ✅ McpToolRegistry | ❌ — servers started but results not consumed |
| `memory_server` | ✅ MemoryServer | ❌ — memory operations bypass the server wrapper |
| `todo` | ✅ TodoList | ❌ — never queried in runtime loop |
| `router` | ✅ ModelRouter (only if pool non-empty) | ❌ — `select()` never called |
| `approval` | ✅ ApprovalManager | ❌ — HITL flow never checked |
| `budget_snapshot` | ✅ BudgetSnapshot | ❌ — never updated mid-session |

**Impact**: Hooks, memory MCP, model routing, HITL, and todos are initialized but **not wired into the REPL or query execution loop**. They are effectively dead infrastructure.

**Suggested fix**: Wire each component into the main agent loop (`run_repl_with_options`), or remove the `#[allow(dead_code)]` suppression and resolve individually.

`#[allow(dead_code)]` on the `StatusBar` impl block means numerous public setters are never called after initialization:

**Suggested fix**: Wire `StatusBar` setters into the REPL event loop after each Claude response, or remove the unused methods.

### 4. README oversells model router — pool is empty by default

**File**: `README.md` (Model Router section)

The README shows a detailed `modelPool` JSON config with capability scoring and claims "Dynamic model selection per query." However:

- The default `settings.json` created by `ante init` has **no model pool entries**
- `agent_sdk/src/router.rs:select_with_fallback()` returns `None` when pool is empty
- `main.rs:372-376` explicitly sets `router: None` when `router_entries.is_empty()`
- The PRODUCTION_AGENT_DEPLOYMENT_NOTES confirms: *"the configured model pool is empty by default, so it is inactive unless settings provide models"*

**Additionally**: The README example JSON uses `"modelId"`, `"capabilityScore"`, `"costPer1kInput"` (camelCase) but the actual `Settings` schema in `protocol-shape/src/settings.rs` uses `"model_id"`, `"capability_score"`, `"cost_per_1k_input"` (snake_case). The example JSON **will not parse** if copied verbatim.

**Suggested fix**:
- Add a note that the model pool is empty by default and the router is inactive until configured
- Add model pool entries to the default `settings.json` generated by `ante init`
- Fix the JSON key casing mismatch between README and schema

### 7. Init symlink logic has fragile parent-path assumption

**File**: `crates/agent-sdk/src/init.rs:71-74`

```rust
let wiki_memory_path =
    default_ante_dir() // ~/.ante
        .parent()      // ~  ← assumes ~/.ante is directly under HOME
        .map(|p| p.join("code").join("wiki-memory").join("wiki"));
```

If `ante_dir` is overridden in settings (e.g. to `/custom/path/.ante`), `.parent()` could return `/custom/path` instead of `HOME`, making the wiki-memory lookup point to `/custom/path/code/wiki-memory/wiki` (wrong location).

**Suggested fix**: Resolve wiki-memory path from `HOME` env var directly instead of deriving from `ante_dir`.
